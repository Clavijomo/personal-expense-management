# AI Rules

Rules and conventions for AI assistants working on this codebase.

---

## General behavior

- Do not make changes without explicit approval. Always explain what you are about to do and wait for confirmation before proceeding.
- Work incrementally. One feature or fix per step — do not batch unrelated changes into a single commit.
- If a decision has tradeoffs, present the options briefly and let the developer choose.
- Do not over-engineer. Implement only what is asked. No extra abstractions, no speculative helpers, no unsolicited refactors.
- Do not add comments or docstrings to code that wasn't changed.

## Git workflow

- All work happens on **feature branches** created from `develop`.
- Branch naming: `feature/`, `fix/`, `docs/`, `chore/` prefixes.
- Commit messages follow **Conventional Commits**: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`.
- Do not include `Co-Authored-By` lines in commit messages.
- Do not create Pull Requests — the developer handles PRs manually.
- Never push directly to `develop` or `main`.
- Never use `--no-verify` or bypass pre-commit hooks.

## Code conventions

- Language: **TypeScript** everywhere. No `any`, no untyped function parameters.
- All UI-facing text is in **Spanish**.
- Use the `@/*` path alias for all internal imports (not relative paths like `../../`).
- Do not modify files inside `src/components/ui/` — those are generated by shadcn/ui.
- Mark components with `"use client"` only when they use browser APIs, hooks with state, or event handlers. Server components are the default.

## State management

- Global state lives in `src/store/useExpenseStore.ts` (Zustand). Only persist what needs to survive page reloads.
- Ephemeral UI state (filters, modal open/close, selected item) stays in local component state — do not add it to Zustand.
- Derive computed values in custom hooks using `useMemo`, not in components or the store.

## Forms

- All forms use `react-hook-form` + `zodResolver`.
- Define Zod schemas in `src/schemas/`. Do not inline schemas inside components or hooks.
- Use `z.number()` with `valueAsNumber: true` in the `register` call for number inputs — do not use `z.coerce.number()` as it causes type conflicts with react-hook-form.
- Wrap shadcn `Select` with `Controller` from react-hook-form. It does not expose a native input ref.

## Components

- Follow the existing layer structure:
  - `src/components/ui/` — shadcn primitives (read-only)
  - `src/components/shared/` — app-wide reusable components
  - `src/components/dashboard/` and `src/components/expenses/` — feature-specific
- Components should be presentational. Business logic belongs in hooks.
- Do not create new components for one-off use cases — inline the markup instead.

## Hooks

- One hook per concern. Do not merge unrelated logic into a single hook.
- Place all custom hooks in `src/hooks/`.
- Hooks that read from the Zustand store must not produce side effects — use `useMemo` for derived data.

## Testing

- Tests live in `src/__tests__/`, mirroring the `src/` structure under `components/` and `hooks/`.
- Use `jest.mock("@/store/useExpenseStore")` to mock the store in hook tests. Do not hit real storage in unit tests.
- Do not mock `window.location` or browser globals — test only what the component renders, not browser behavior.
- Each test file should cover one unit (one component or one hook).

## What to avoid

- Do not use `git add .` or `git add -A` — stage files explicitly by name.
- Do not use `--amend` on commits that have already been discussed or shared.
- Do not create `README` or documentation files unless explicitly requested.
- Do not add `console.log` statements to production code.
- Do not install new dependencies without asking first.
